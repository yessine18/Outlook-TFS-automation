Chapter 1 — Problem Statement

The Operational Bottleneck

Manual email triage creates delays, inconsistencies, and zero knowledge reuse

❌ Current Pain Points

  • High latency — manual reading, classification, and routing of every support email
  • No traceability — requests buried in shared inboxes with no audit trail
  • Duplicate effort — same known issues resolved repeatedly by L1 engineers
  • Inconsistent communication — no standardized feedback to end-users
  • Zero knowledge capitalization — past resolutions are never reused

✅ Proposed Solution

  • Build an end-to-end "zero-touch" AI orchestrator
  • Transform emails into structured, trackable ADO work items automatically
  • Deploy an Agentic AI capable of autonomously resolving L1 tickets
  • Implement thread-aware communication and client feedback loops
  • Provide real-time monitoring via a live dashboard

Chapter 1 — Theoretical Foundations

Core Concepts & Technologies

🤖
Large Language Models (LLMs)
Transformer-based generative AI for intelligent text comprehension and structured metadata extraction. We use Groq / LLaMA 3.3 70B.
🗄️
RAG (Retrieval-Augmented Generation)
Grounds AI responses in verified knowledge bases using vector embeddings (384D, all-MiniLM-L6-v2) and cosine similarity search via pgvector.
🧠
Agentic AI (ReAct Framework)
Autonomous AI agents capable of multi-step reasoning and dynamic tool selection via LangGraph create_react_agent.
🔌
Model Context Protocol (MCP)
Open standard (Anthropic) for securely connecting LLMs to external APIs without exposing credentials to the AI model.
⚡
Event-Driven Architecture
Background daemon polling, REST webhooks, and async inter-system communication via Microsoft Graph and Power Automate.
🔐
OAuth 2.0 Client Credentials
Daemon-based authentication via Azure Entra ID for zero-human-intervention access to Microsoft Graph APIs.

Chapter 4 — Evolutionary Prototyping

Iterations 1 – 3: Foundation to Resilience

Iteration 1 — Basic API Integration

.NET Worker Service • Microsoft Graph SDK v5 • Azure DevOps REST API

Background daemon polls Outlook every 60s. Creates ADO Issue from raw email subject/body. Marks email as read.

⚠️ Limitation: Raw HTML dumped into ADO. No metadata extraction, no routing, no severity classification.

Iteration 2 — LLM-Powered Extraction & Routing

Groq API (LLaMA 3.3 70B) • CsvHelper • JSON Prompt Engineering

LLM extracts structured metadata (coreProblem, severity, jobField). CSV-based routing to correct assignee. HTML auto-reply emails with QR codes. Domain whitelist spam filter.

⚠️ Limitation: No persistent storage. If pipeline crashes after LLM but before ADO — data is lost forever.

Iteration 3 — State Management & Fault Tolerance

PostgreSQL • Entity Framework Core • PipelineStatus State Machine • TicketStateLogs

"Database-First" persistence. 10-value state machine. Bidirectional ADO sync via WIQL. Step-level error isolation. TMA alert emails on failures.

⚠️ Limitation: Every ticket escalated to humans. Known issues that could be auto-resolved still waste L1 engineering hours.
Chapter 4 — Evolutionary Prototyping

Iterations 4 – 6: Intelligence to Autonomy

Iteration 4 — Agentic AI & Auto-Resolution

Python–C# Bridge • LangGraph Orchestrator • RagVerdict JSON • pgvector

Passes problem to Python LangGraph agent with 3 tools. If AI finds solution (confidence > 70%), injects green solution box into auto-reply with Accept/Reject buttons. ADO Work Item always created regardless.

⚠️ Limitation: No feedback loop. System delivers AI solutions but can't verify if they actually helped the client.

Iteration 5 — Client Validation & Feedback

Minimal API Webhooks • Outlook Actionable Messages • Adaptive Cards v1.4

Accept → ADO set to "Done". Reject → stays open for human. Closure emails embed star rating (1–5) Adaptive Card. Feedback persisted to DB + ADO discussion.

⚠️ Limitation: All replies are standalone emails. Client follow-ups create duplicate ADO tickets. No thread tracking.

Iteration 6 — Thread Intelligence & Full Integration

Graph .Reply • ConversationId Tracking • LLM Summarization • Teams Webhooks • Dashboard

Thread-aware .Reply endpoint. ConversationId matching prevents duplicates. LLM summarizes follow-ups as ADO comments. Color-coded Teams cards (🟢/🟠). Live dashboard SPA.

Chapter 3 — AI Orchestration

Agentic RAG Architecture

Three autonomous tools orchestrated by a LangGraph ReAct agent

🗄️
Tool 1: Internal KB
pgvector cosine similarity search against 14,000+ Microsoft Docs embedded as 384D vectors. Returns Top-3 most relevant documentation chunks.
📚
Tool 2: Learn Catalog MCP
Custom MCP server we built. Authenticates via Entra ID to fetch official Microsoft training modules, learning paths, and certifications.
🌐
Tool 3: Official MS MCP
Custom HTTP wrapper bypassing Microsoft's 405 error. Parses SSE streams from the official learn.microsoft.com/api/mcp endpoint.
Key Innovation: Microsoft's official MCP server lacks training/certification content. We built our own custom MCP server (learn_catalog_mcp.py) to bridge this gap — effectively engineering the feature Microsoft hasn't released yet.

Key Design Patterns

💾 Database-First Persistence
Ticket saved to PostgreSQL BEFORE ADO creation. If ADO fails, data is safe.
🏷️ Tag-Based Email Dedup
EmailSent_Done / EmailSent_Doing tags prevent duplicate notification emails across polling cycles.
🔄 Scoped DbContext
IServiceScopeFactory creates fresh DB contexts inside the Singleton background service.
👻 Hidden Metadata
Sender info stored as invisible data-* HTML attributes in ADO descriptions for state-change email routing.
